# RULES.md — AI向け絶対遵守コーディング規約

`docs/STYLEGUIDE.md`（コーディング規約）・`docs/ARCHITECTURE.md`（構成）を**補完**する、AI（Claude Code 等）に特化した厳格ルール。
`CLAUDE.md` の「最重要ルール」と矛盾する場合は **CLAUDE.md が優先**。本書はそれをAIの振る舞いレベルに落とし込む。

> 立ち位置：CLAUDE.md＝プロジェクト運用ルール／STYLEGUIDE＝人間にも共通のスタイル／**RULES.md＝AIが暴走・幻覚・破壊しないためのガードレール**。

---

## 0. 大原則（この3つを常に念頭に）

1. **推測で書かない。不明なら止めて聞く**（§2 ハルシネーション対策）。
2. **既存を壊さない。壊す前に影響範囲を出して許可を得る**（§3 破壊的変更の防止）。
3. **プライバシーとSEOの不変条件を破らない**（バックエンド無し・ブラウザ内処理・SEO3点セット。CLAUDE.md ルール8・9）。

## 1. コード品質

### 1.1 型（TypeScript）
- **型推論に丸投げしない**：公開関数・フックの**戻り値／引数・props は明示的に型注釈**する。特に `_hooks` の返り値と `_components` の props。
- **`any` 禁止**。避けられない外部境界（Three.js 等の緩い型）では `unknown` ＋絞り込み、またはローカルな最小型を定義する。理由をコメントで一言残す。
- **`as`（型アサーション）は最終手段**。使うなら「なぜ安全か」を1行コメント。`as any` は不可。
- **`_types.ts` を単一ソースに**：ツール／記事のデータ型は既存の `_types.ts` を再利用・拡張する。同じ形の型を各所で再定義しない。
- **`npx tsc --noEmit` が通ること**を型変更時の必須確認とする（STYLEGUIDE §8）。

### 1.2 痕跡を残さない
- **`console.*` 禁止**：コードベースの `console.*` は**0件を維持**（DEBUGGING.md §4）。デバッグ出力はコミット前に必ず除去。
- **コメントアウトした死コードを残さない**：不要コードは消す（git が履歴を持つ。復元は `git` で行う）。
- **未使用の import / 変数 / 到達不能コードを残さない**（`npm run lint` の warning も新規に増やさない）。
- **TODO を安易に埋め込まない**：本当に必要なら `MEMORY.md` §4 に書く（コード内 TODO は腐る）。

### 1.3 コメント方針
- コメントは**「なぜ」**を書く（「何を」はコードで表現する）。
- 周囲のコードの**コメント密度・命名・イディオムに合わせる**。過剰なコメントを足さない。
- ユーザー向け文言のコメントではなく、実装意図・落とし穴（ハイドレーション・物理演算の非決定性など）を残す。

### 1.4 スタイル遵守（STYLEGUIDE の再掲・違反しがちな点）
- **色をハードコードしない**：`text-ck-*` / `var(--ck-*)` / `ck.*` のみ。例外は描画用途（SVG/Canvas・`next/og`）だけ（STYLEGUIDE §2）。
- **ユーザー向け文言は必ず日英両対応**（`translations.ts` か `locale === 'ja' ? … : …`）。片方だけ追加しない。
- **非公開モジュールは `_` プレフィックス**。フックは `useXxx`、コンポーネントは PascalCase。
- **共通化を優先**：`ToolHeader`/`ToolFooter`・既存ユーティリティクラス（`.ck-*`）・`_lib` ヘルパーを再実装しない。

## 2. ハルシネーション対策（最重要・AI固有）

**「たぶんこうだろう」で実装しない。CastKit は破壊的変更を含む Next.js を使っており、記憶に頼ると必ず間違える。**

1. **Next.js 固有のコードを書く前に、必ず `node_modules/next/dist/docs/` の該当ガイドを読む**（AGENTS.md／CLAUDE.md ルール1）。バージョンは `next@16.2.7`、React 19。訓練データの API と違う前提で確認する。
2. **未定義の API・ライブラリ関数を推測で呼ばない**。使う前に:
   - プロジェクト内なら該当ファイルを **Read/Grep で実在確認**（署名・引数・戻り値）。
   - ライブラリなら `node_modules` の型定義かドキュメントで確認。
   - 確認できないものは**呼ばない**。
3. **仕様が不明確・複数解釈できる・トレードオフがある場合は、勝手に決めずユーザーに質問を返す**。質問は `MEMORY.md` §5 にも積む。特に:
   - 新しい依存を足す判断（CLAUDE.md ルール10）。
   - i18n 方式（パスベース移行など）・SEO 構造・データ永続の設計変更。
   - 「どちらの UX が正か」がユーザーの好みに依存する箇所。
4. **メモリ（recalled memory）や本書の記述が古い可能性を疑う**：ファイル名・関数・フラグに言及するメモリは、**実在をコードで再確認してから**推奨・使用する。
5. **存在しない skill / コマンド / 設定を案内しない**。確証が無いものは「確認が必要」と明示する。

> 迷ったときのデフォルトは**「実装より確認」**。手戻り1回のほうが、誤った自信で書いたコードより高くつく。

## 3. 破壊的変更の防止

### 3.1 大きく書き換える前に
既存コードを**大幅に書き換える・共通コンポーネントの I/F を変える・複数ファイルに波及する**場合は、着手前に:

1. **影響範囲を Grep で洗い出して提示**（どのツール／ページ／記事が該当するか）。
2. **変更方針と代替案を1〜2行で示し、許可を得る**。
3. 許可後に着手。**勝手にリファクタして「ついでに整えました」を大量に混ぜない**。

### 3.2 コミットを安全網にする（CLAUDE.md ルール7・開発の進め方）
- **新ツール／大きめの変更はブランチを切る**：`git switch -c tool/<名前>`。`main` を直接いじらない。
- **意味のある単位でコミット**（1コミット1関心事）。「骨組み」と「アニメ」は別コミット。
- **`git reset --hard` / `git checkout -- <file>` など取り消しづらい操作の前は、必ず `git log` を見せ、どこまで戻すかを確認してから実行**。勝手に巻き戻さない。
- ユーザーが「さっきのは無し」と言ったら **`git revert`（履歴を残す安全な戻し方）** を基本にする。

### 3.3 外部・非可逆な操作は確認してから
- **push・ブランチ作成・削除・本番に出る変更**は、ユーザー指示か区切りタイミングでのみ行う。
- 削除・上書き対象は**事前に中身を見る**。説明と食い違う・自分が作ったものでないなら、進めず**その相違を報告**する。

### 3.4 不変条件（壊したら重大）
- **バックエンド無し・ブラウザ内処理を維持**。ユーザー入力をサーバー／外部に送信しない（CLAUDE.md ルール9）。
- **広告・解析以外の外部スクリプト／外部 API を安易に足さない**（必要時は理由を明記して相談）。
- **新ページには SEO 3点セット必須**（metadata＋`pageAlternates`＋JSON-LD。CLAUDE.md ルール8）。
- **新ツール追加は `docs/NEW_TOOL.md` のチェックリスト**を漏れなく（sitemap/ホーム/AppShell/RelatedTools/guide/TOOL_SHARE/i18n の手動登録）。
- **依存を安易に増やさない**（CLAUDE.md ルール10）。標準機能・既存ライブラリ（`marked`/`sonner` 等）で代替できないか先に検討。

## 4. 環境構築・シークレット

- **APIキー・シークレットを絶対にハードコードしない**。そもそも本プロジェクトはバックエンド無しで秘匿情報を持たない設計。新たに必要になったら**まず「本当に必要か（プライバシー不変条件を壊さないか）」を相談**する。
- **公開値**（`NEXT_PUBLIC_SITE_URL` 等）は環境変数経由（Vercel 設定）。`NEXT_PUBLIC_*` はビルド時に埋め込まれる＝**機密を入れない**（詳細は `docs/DEPLOY.md`）。
- **`.env*` をコミットしない**（`.gitignore` 前提）。値をコード・ドキュメント・コミットメッセージに書かない。
- **`node_modules` / `.next` を git に含めない**（`.gitignore` 済み）。

## 5. 検証（完了の定義）

変更を「完了」と呼ぶ前に、次を満たす（DEBUGGING.md §5 と一致）。

- [ ] **`npm run lint`** が exit 0（0 errors・新規 warning を増やさない）
- [ ] **`npm run build`** 成功（型チェック込み。型変更時は `npx tsc --noEmit` も）
- [ ] **日英・モバイル幅・（本番限定なら）`npm run build && npm run start`** で該当挙動を確認
- [ ] **回帰確認**：他ツール／他ページ／もう一方のロケールを壊していない
- [ ] **一時コード・`console.*` を残していない**
- [ ] **仕様・構成・規約に影響したら対応 `docs/` を更新**（CLAUDE.md ルール6）
- [ ] **`MEMORY.md` を更新**（完了タスクの移動・次の一手の記録）

> 報告は正直に：テストやビルドが落ちたらその出力とともに「落ちた」と言う。スキップした手順は「スキップした」と言う。「たぶん動く」を「動いた」と言わない。
