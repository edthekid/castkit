# SECURITY.md — セキュリティ要件とベストプラクティス

CastKit のアーキテクチャ（**バックエンド無し・SSG・ブラウザ内処理**）で現実的に想定される脅威と、その防止方針。

> 大前提：本サイトは**サーバーサイドのアプリケーションロジック・DB・ユーザー認証を持たない**。
> したがって「一般的な Web アプリのセキュリティ」の多くは**該当しない**。
> 本書は**この静的サイト構成で実際に効く対策だけ**を扱い、存在しない攻撃面を守る空論は書かない。

---

## 1. 脅威モデル（この構成で守るべきもの）

| 攻撃面 | 本サイトでの該当 | リスク |
|--------|------------------|--------|
| サーバー認証・セッション | **無し**（ログイン機能が存在しない） | 該当せず |
| SQL インジェクション | **無し**（DB が存在しない） | 該当せず |
| サーバー側 API | **無し**（バックエンド無し） | 該当せず |
| **XSS（DOM/格納型）** | **有り**：記事の Markdown→HTML、ユーザー入力の画面表示、localStorage 復元 | **主要リスク** |
| **サプライチェーン**（npm 依存） | **有り**：`three`/`cannon-es`/`marked`/`daisyui` 等 | 中 |
| **サードパーティスクリプト**（AdSense・Analytics） | **有り**：外部 JS を `<head>` に読み込む | 中（レビュー必須） |
| **ホスティング/DNS**（Vercel・Cloudflare） | **有り**：ドメイン・TLS・リダイレクト | 低〜中 |
| **プライバシー漏洩**（入力の外部送信） | **有り**：不変条件を破ると発生 | **重大**（設計不変条件） |

**結論**：守る中心は **① XSS ② サプライチェーン ③ プライバシー不変条件の維持**。認証・DB・CORS の設計は不要（存在しないため）。

---

## 2. 認証・認可・セッション

**該当なし（意図的）。**

- CastKit に**ログイン・アカウント・保護ルートは存在しない**。全ページが公開・静的。
- **セッション管理・トークン（Cookie/LocalStorage いずれも）を導入しない**のが現在の設計。
- ユーザーの設定（カスタムお題・スコア・ダイス設定等）は **`localStorage` に保存するが、これは「機密」ではなく「利便性データ」**。ログイン資格情報ではない。したがって:
  - localStorage には**個人を特定する情報・秘匿情報を入れない**（現状も入れていない）。
  - トークンや資格情報を将来もし扱うなら、それは**アーキテクチャの根本変更**であり、`RULES.md` §3 に従い**着手前にユーザーの明示的許可**を得る（安易にバックエンド／認証を足さない）。

> もし将来「認証が要る機能」を求められたら、まず**それがプライバシー不変条件（§5）とバックエンド無し方針を壊す**ことを説明し、代替（クライアント完結で実現できないか）を提示すること。

---

## 3. データ保護（XSS 対策 ＝ 最重要）

DB が無いため **SQL インジェクションは非該当**。代わりに **XSS を最重要対策**とする。

### 3.1 記事（Markdown → HTML）
- 記事本文は `marked` で**ビルド時（サーバー側）に HTML 化**し、`ArticleLayout` が `dangerouslySetInnerHTML` で描画する（`docs/ARCHITECTURE.md`）。
- **安全である根拠＝入力がユーザー由来でなく、リポジトリ内の信頼できる著者（＝我々）が書いた Markdown だから**。第三者が本文を投稿する経路は無い。
- **禁止事項**：
  - 記事本文に**外部から取得した文字列・ユーザー入力を混ぜない**（信頼境界を崩す）。
  - `marked` の設定で**生 HTML を許可したうえに未検証の入力を通す**構成にしない。
  - もし将来「ユーザー投稿・外部フィード」を HTML 化するなら、**サニタイズ必須**（例：`DOMPurify` 相当）。ただし依存追加＝ CLAUDE.md ルール10 に従い相談。

### 3.2 ユーザー入力の画面表示（各ツール）
- チーム名・候補・お題・スコア等の入力は **React が既定でエスケープして描画**する（`{value}` は自動エスケープ）。**これを維持する**。
- **`dangerouslySetInnerHTML` をユーザー入力に対して使わない**（記事の信頼済み HTML 以外では原則使用禁止）。
- 入力を URL・`href`・`style`・属性に流し込むときは注意（`javascript:` スキーム・属性インジェクション）。ユーザー入力を**そのまま `href` にしない**。
- 数値入力（スコア・ダイス個数/面数）は**範囲・型を検証**してから使う（NaN・巨大値・負数でロジックや DOM を壊さない）。

### 3.3 localStorage の復元
- 復元時は**信頼できない可能性のあるデータ**として扱う（ユーザーが手動で書き換え可能）。
  - `JSON.parse` は **try/catch** で囲み、壊れた値でクラッシュさせない。
  - 復元値を**そのまま描画・計算に使う前にスキーマ検証／サニタイズ**（想定外の型・キーを弾く）。
- localStorage の値を **`dangerouslySetInnerHTML` や `eval` 相当に渡さない**。

### 3.4 CORS
- **サーバー API が無いため CORS 設定は不要**。外部への `fetch` を新たに足さないのが原則（§5）。
- どうしても外部リソースを読む場合のみ、その必要性・送信内容を明記して相談する。

### 3.5 CSP・セキュリティヘッダ（推奨・任意）
- 静的サイトでも **Content-Security-Policy / `X-Content-Type-Options: nosniff` / `Referrer-Policy` / `X-Frame-Options`（または CSP `frame-ancestors`）** を付けると XSS・クリックジャッキング耐性が上がる。
- Vercel では `next.config` の `headers()` か Vercel の設定で付与可能。**AdSense・Vercel Analytics の配信ドメインを許可**する必要があるため、CSP は「まず report-only で検証 → 本適用」の順で慎重に。**導入時はユーザーに影響（広告表示が壊れないか）を確認**してから。

## 4. シークレット管理・環境変数

- **秘匿情報をコード・リポジトリ・コミットメッセージに書かない**（`RULES.md` §4）。
- 現在の環境変数は **`NEXT_PUBLIC_SITE_URL`（= `https://cast-kit.com`）のみ**で、これは**公開値**（canonical/sitemap/OGP/シェアURL の基準。`docs/DEPLOY.md`）。
  - **`NEXT_PUBLIC_*` はビルド時にクライアントへ埋め込まれる**＝**機密を絶対に入れない**。秘密が必要な値には `NEXT_PUBLIC_` 接頭辞を付けない。
- **`.env*` は Git 管理外**（`.gitignore` 前提）。ローカルの `.env.local` に秘密を置いてもコミットしない。
- **CI/CD（Vercel）**：環境変数は **Vercel の Project Settings → Environments** で管理する。
  - 公開 URL のような非機密は Sensitive オフ・値は Production に設定（`docs/DEPLOY.md` §2-3）。
  - もし将来トークン等の**機密**を扱うなら **Sensitive をオンにし、`NEXT_PUBLIC_` を付けない**（サーバー専用に留める）。ただし本サイトはサーバー処理を持たないため、そもそも機密が要る設計を避ける。
- **AdSense 審査コード・アソシエイト ID（`castkit-22`）** は公開情報であり秘匿対象ではない（HTML/リンクに出る前提のもの）。混同して「隠すべき」扱いにしない。

## 5. プライバシー不変条件（設計上のセキュリティ要件）

CastKit の中核的な約束＝**ユーザー入力をサーバー／外部に送信しない**（CLAUDE.md ルール9・`/privacy` の開示）。これは機能であると同時に**セキュリティ要件**。

- ツールの入力（メンバー名・お題・スコア等）は**ブラウザ内のみで処理**し、**外部へ送らない**。
- **広告（AdSense）・解析（Vercel Analytics）以外の外部スクリプト・外部 API・トラッカーを足さない**。追加が必要なら**理由・送信内容・プライバシー影響を明記して相談**。
- 解析は **Vercel Analytics（Cookieレス）** を使用。個人を特定する送信をしない。
- `/privacy` の開示内容（localStorage 利用・AdSense/解析の Cookie）と**実装を一致させ続ける**。実装を変えたら privacy ページも同じ作業で更新（CLAUDE.md ルール6）。

> この不変条件を破る変更（入力の外部送信・新規トラッカー・バックエンド追加）は、**単独判断で実装せず必ずユーザーの許可を得る**（`RULES.md` §3.4）。

## 6. 依存関係（サプライチェーン）

サーバーが無い分、**クライアントに配られる npm 依存が主要な攻撃面**（悪意ある更新・既知脆弱性）。

### 6.1 運用
- **`npm audit`** を定期的に実行し、**High/Critical を優先対応**（`npm audit fix`、破壊的更新が必要なら手動で検証）。
- **依存追加は最小限**（CLAUDE.md ルール10）。追加時は:
  - 標準機能・既存依存（`marked`/`sonner`/`three` 等）で代替できないか先に検討。
  - 追加するライブラリの**メンテ状況・ダウンロード数・依存の重さ**を確認。
  - **クライアントに載る依存か devDependencies か**を区別（テスト系＝dev はバンドル・SEO に影響しない）。
- **バージョン更新（特に Next/React/Tailwind のメジャー/マイナー）は `docs/UPGRADING.md` の手順と検証チェックリストに従う**（破壊的変更の温床）。
- **lockfile（`package-lock.json`）をコミット**し、再現性を確保。身に覚えのない依存差分が出たら止めて確認。

### 6.2 サードパーティ JS（AdSense・Analytics）
- `<head>` に読み込む外部スクリプトは **AdSense・Vercel Analytics に限定**。他を安易に足さない。
- 追加・変更時は**配信ドメイン・読み込み方法（async/defer）・プライバシー影響**を確認する。CSP を導入している場合は許可ドメインの更新を忘れない。

## 7. ホスティング・ドメイン

- **TLS/HTTPS は Vercel が自動提供**。`http` で配信しない（`NEXT_PUBLIC_SITE_URL` も `https`・末尾スラッシュなし）。
- **DNS は Cloudflare**。DNS レコード・所有権確認 TXT（Search Console）を不用意に消さない。
- **`castkit-one.vercel.app` → `cast-kit.com` のリダイレクト整理**（`MEMORY.md` §4 の TODO）は、重複 URL＝SEO 兼「正規オリジンの明確化」の観点でも実施が望ましい。

## 8. インシデント時の初動（この構成での現実的手順）

サーバーが無いため「サーバー侵害」は想定しにくい。現実的な事象と初動:

| 事象 | 初動 |
|------|------|
| 依存の脆弱性公表（High/Critical） | `npm audit` で影響確認 → 更新 → `npm run build` 検証 → デプロイ |
| 不正な外部スクリプト混入の疑い | `git log`・差分で `<head>`/依存の変更を確認 → 身に覚えのない変更を revert |
| 記事/UI に意図しない HTML 実行 | `dangerouslySetInnerHTML` の入力源をたどり、信頼境界外の入力が混ざっていないか確認 → 遮断・サニタイズ |
| ドメイン/DNS 異常 | Cloudflare DNS と Vercel Domains 設定を確認（TXT・CNAME・リダイレクト） |
| 誤って秘密をコミット | 直ちに該当値を無効化（ローテーション）→ 履歴からの除去を検討（`git` 履歴書き換えは影響大につきユーザー確認の上で） |
