# TESTING.md — テスト戦略と実装ガイドライン

CastKit の品質保証の指針。**新しいロジック／コンポーネントを書いたら、この方針に沿ってテストも用意する。**

> 前提：本プロジェクトは **バックエンド無し・SSG・ブラウザ内処理**（`docs/ARCHITECTURE.md`）。
> テスト対象は「①純粋ロジック（抽選・判定・整形）②React フック／状態 ③ハイドレーション・i18n・SEO の壊れ検知」に収束する。サーバー API・DB・認証は存在しないため、それらのテストは不要。

---

## 0. 現状（2026-07-04 時点）

- **テストフレームワークは未導入**（`package.json` に test 依存なし。`npm run test` も未定義）。
- 現在の品質ゲートは **`npm run lint` ＋ `npm run build`（型チェック込み）** のみ（CLAUDE.md ルール2）。
- 本ドキュメントは「これから入れるなら何を・どう入れるか」を定義する**導入ガイド兼運用ルール**。
  実際に依存を追加する前に、CLAUDE.md ルール10（依存を安易に増やさない）に従い **導入の是非をユーザーに一度確認する**。

## 1. テストスタック（推奨）

| レイヤー | フレームワーク | 採用理由 |
|----------|----------------|----------|
| **単体（ユニット）** | **Vitest** | Vite/ESM ネイティブで **Turbopack + TS + Tailwind v4** 構成と噛み合う。Jest 互換 API で学習コスト低。速い。設定が最小。 |
| **コンポーネント/フック** | **Vitest + @testing-library/react + jsdom** | React 19 対応。`useXxx` フックと `_components` を実ブラウザ無しで検証。日英切替・localStorage 復元の確認に十分。 |
| **E2E** | **Playwright** | SSG の**本番ビルド出力**（`next build && next start`）に対して、ハイドレーション・アニメーション・localStorage 永続を実ブラウザで検証。クロスブラウザ・モバイル幅・`?lang=en` を1つのランナーで回せる。 |

### 依存追加の注意（重要）
- 上記はすべて **`devDependencies`**。クライアントバンドル・SEO・表示速度に**影響しない**（CLAUDE.md ルール10の懸念に該当しない）ことを PR 説明で明記する。
- 3D サイコロ（Three.js + cannon-es）の**物理挙動そのものは E2E/ユニットの対象外**（非決定的）。テストするのは「判定・整形・履歴フォーマット」などの**決定論的な純粋関数**に限る（後述 §4）。

### 想定スクリプト（導入時に `package.json` へ追加）
```jsonc
"scripts": {
  "test": "vitest run",
  "test:watch": "vitest",
  "test:e2e": "playwright test"   // 事前に next build && next start、または webServer 設定
}
```

## 2. 記述ルール

### 2.1 テストの粒度（優先順位）
1. **純粋ロジック（最優先・費用対効果が最大）**
   抽選・シャッフル・チーム分けの全組み合わせ生成・チンチロ役判定・ダイス整形・Markdown 前段など、**入力→出力が決定的な関数**。乱数は seed 注入か「分布・不変条件」で検証（§4）。
2. **フック／状態**
   `useXxx` の状態遷移（アンドゥ、順位並べ替え、localStorage 復元、モード切替）。
3. **i18n・SEO・ハイドレーションの壊れ検知（E2E 少数）**
   代表 1〜2 ツールで「ja/en 切替が反映」「`?lang=en` でも hydration 警告ゼロ」「canonical/JSON-LD が出力される」を担保。

> ピラミッド比率の目安：**ユニット 70% / フック 20% / E2E 10%**。E2E は重いので「壊れたら致命的な導線」だけに絞る。

### 2.2 モック戦略
- **原則モックしない**：純粋関数はそのまま呼ぶ。過剰なモックは実装の写経になり価値が下がる。
- **`Math.random` は「モック」ではなく「注入 or 統計検証」**：
  - できれば関数が乱数生成器を引数で受け取る形（`rng = Math.random`）にして、テストでは固定 rng を渡す。
  - 難しければ N 回試行して **不変条件**（合計が保存される／全員が必ず1チームに属す／重複が無い／範囲 1〜100）を検証する。
- **`localStorage`**：jsdom の実装をそのまま使い、各テストで `localStorage.clear()` を `beforeEach` で行う。専用モック不要。
- **`window` / `matchMedia` / `ResizeObserver`**：jsdom に無い API はテストセットアップで最小スタブ。
- **Three.js / cannon-es / `next/dynamic(ssr:false)`**：**ユニットではロードしない**。3D コンポーネントは E2E 側で「クラッシュせず描画が始まる」だけ確認し、出目の値はロジック関数側でテストする。
- **`next/og`（OGP・アイコン）**：画像バイナリの内容はテストしない。ビルドが通ること（`npm run build`）で担保。

### 2.3 カバレッジ目標
- **`app/**/_utils.ts` / `_constants` 由来の純粋ロジック・判定エンジン（例 `app/dice/_chinchiro.ts`）：行 90%以上**（ここが本体）。
- **`_hooks`：主要な状態遷移パスを網羅**（数値カバレッジより「アンドゥ・リセット・復元」等の代表シナリオ充足を優先）。
- **描画専用（SVG/Canvas 座標計算、Three.js ラッパ）：数値目標を課さない**（非決定・視覚要素のため E2E とスナップショット目視で担保）。
- 全体の行カバレッジを単一の関門にはしない（SSG サイトでは意味が薄い）。**「壊れたら痛い所が緑」**を基準にする。

## 3. ディレクトリ構造・命名規約

### 3.1 配置：**コロケーション（同居）を基本**
本プロジェクトは既に `_hooks` / `_utils` / `_components` を**機能フォルダに同居**させている（`docs/ARCHITECTURE.md` §3）。テストも同じ思想で**対象ファイルの隣**に置く。

```
app/dice/
├─ _chinchiro.ts
├─ _chinchiro.test.ts        # ← 同居
├─ _diceShapes.ts
├─ _hooks/
│   ├─ useDice.ts
│   └─ useDice.test.ts       # ← 同居
└─ _components/
    ├─ DiceHistory.tsx
    └─ DiceHistory.test.tsx  # ← 同居
```

- **E2E だけは分離**：リポジトリ直下 `e2e/` に置く（実ブラウザ・本番ビルド対象で性質が違うため）。
  ```
  e2e/
  ├─ smoke.spec.ts           # 全ツールが 200 で開く・hydration 警告ゼロ
  ├─ i18n.spec.ts            # ?lang=en 切替
  └─ scoreboard.spec.ts      # localStorage 永続・アンドゥ
  ```

### 3.2 命名
- ユニット/フック/コンポーネント：**`<対象>.test.ts(x)`**（対象と同名＋`.test`）。
- E2E：**`<機能>.spec.ts`**（`e2e/` 配下）。
- `_` プレフィックス規約（STYLEGUIDE §6）は維持。テストは**非公開モジュール同様に扱い**、`.test.ts` は Next のルーティング対象にならない（ページは `page.tsx` のみ）。

### 3.3 除外設定
- `tsconfig` / eslint がテストを拾うよう型は通す一方、**本番ビルドに混ぜない**。Vitest は Next のビルドと独立して動くため通常は問題ないが、`next build` が `*.test.*` を巻き込まないことを導入時に確認する。

## 4. このプロジェクト特有のテスト観点（チェックリスト）

各ツールで「決定論的に検証できる不変条件」を優先的にテストする。

| ツール | 純粋ロジックで検証すべき不変条件 |
|--------|----------------------------------|
| **チーム分け** | 全員が必ず1チームに属す／人数差が最小（均等）／固定メンバーは指定チーム固定／全組み合わせ生成に重複が無い |
| **ルーレット** | 候補が空でも落ちない／オートムーブで当選者が次回候補から除外される |
| **あみだ** | 参加者数＝結果数のとき全単射（1本1結果）／線をたどった結果が整合 |
| **お題ガチャ** | カテゴリ絞り込みが効く／カスタムのみモードで default を混ぜない／localStorage 復元 |
| **ディベート** | 陣営がランダムに二分される／`テーマ,陣営A,陣営B` のパースが堅牢（区切り欠落・全角） |
| **スコアボード** | 加減算が正しい／アンドゥで直前値に戻る／順位ソートが安定／リセットの範囲（スコアのみ/全体）が正しい／localStorage 復元 |
| **サイコロ** | **チンチロ役判定（`_chinchiro.ts`）＝役・倍率・強さの網羅**（ピンゾロ/ゾロ目/シゴロ/通常/ヒフミ/目なし）／**1d100 が 1〜100 かつ `00`＋`0`＝100**／Discord 整形 `🎲 2d6 -> [3, 5] = 8` の文字列一致／個数・面数・モードの localStorage 復元 |

**横断（E2E で少数）**
- 全ツールページが **200 で開き、Console に `Hydration failed` が出ない**（SSR=ja 固定と初期描画の一致確認）。
- `?lang=en` で主要文言が英語化される／`&` が接続詞に使われていない（STYLEGUIDE §5）。
- 各ページに `canonical` / `hreflang` / JSON-LD が出力されている（CLAUDE.md ルール8 の回帰防止）。

## 5. AI（あなた）向け実装プロンプト — テスト生成の必須手順

**新しいロジック関数・フック・コンポーネントを作成／変更したら、同じ作業内で以下を行う。**

1. **対象が純粋関数か副作用付きかを判定する。**
   - 純粋（抽選・判定・整形・座標計算のうち決定論的な部分）→ **`<対象>.test.ts` を同居生成**。乱数依存なら rng 注入 or 不変条件テスト（§2.2）。
   - フック（`useXxx`）→ `@testing-library/react` の `renderHook` で**状態遷移＋localStorage 復元**を検証。
   - 純粋部分と副作用が混ざっていたら、**テストしやすいよう純粋関数を `_utils.ts` に切り出してからテストする**（テスト容易性のためのリファクタは歓迎、ただし §CLAUDE ルール7 の粒度でコミット）。

2. **§4 の不変条件表を参照**し、そのツールで「決定論的に真であるべき性質」を最低1つはアサートする。

3. **i18n を含む文言変更をしたら**、ja/en の両キー存在を前提に、英語表示の代表テスト（または E2E の `?lang=en`）を1つ用意する。

4. **テストフレームワーク未導入の段階では**：勝手に Vitest/Playwright を `npm install` しない。代わりに
   - テストコードは書いて同居配置し、
   - 「導入するには `vitest` / `@testing-library/react` / `jsdom`（または `@playwright/test`）を devDependencies に追加する必要がある」旨を**明示してユーザーに確認**する（CLAUDE ルール10）。

5. **絶対にやらないこと**：
   - 3D 物理演算の**出目そのもの**を期待値固定でテストする（非決定的＝フレーキー）。ロジック関数側でテストする。
   - テストのために本番コードへ `console.log` やデバッグ分岐を残す（DEBUGGING.md §4：`console.*` は0件を維持）。
   - ネットワーク・外部 API のモック（そもそも存在しない。増やさない）。

6. **完了条件**：追加/変更したテストが**ローカルで緑**、かつ **`npm run lint` と `npm run build` が通る**こと（DEBUGGING.md §5 の Definition of Done に準拠）。
