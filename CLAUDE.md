# CLAUDE.md — AI 開発ルール

CastKit を AI（Claude Code など）で開発・編集する際のルール。**コードを書く前に、このファイルと `docs/` を必ず参照すること。**

---

## プロジェクト概要

CastKit は、配信・ゲームイベント向けの**無料ツール集**（チーム分け／ルーレット／あみだくじ／お題ガチャ／ディベート）。

- **技術**: Next.js 16（App Router・Turbopack） / React 19 / TypeScript / Tailwind CSS v4 / daisyUI
- **多言語**: 日本語・英語（同一URLでクライアント側切替。パスは分けない）
- **バックエンド無し**: 静的生成（SSG）＋ ブラウザ内処理。サーバーに個人情報を送らない
- **本番**: https://cast-kit.com （Vercel ホスティング / Cloudflare DNS）
- **目的**: 集客（SEO）と広告収入（AdSense）

---

## 最重要ルール（必ず守る）

1. **Next.js は破壊的変更あり**（AGENTS.md 参照）→ Next 固有のコードを書く前に `node_modules/next/dist/docs/` の該当ガイドを読む。
2. **変更したら必ず `npm run lint` と `npm run build` で検証**（lint＋型チェック＋本番ビルドが通ることを確認してから完了とする）。
3. **色をハードコードしない** → デザイントークンを使う：`text-ck-*`（Tailwind）/ `var(--ck-*)`（CSS）/ `ck.*`（インラインstyle）。詳細は `docs/STYLEGUIDE.md`。
4. **ユーザー向け文言は必ず日英両対応**（`app/_i18n/translations.ts` か `locale === 'ja' ? … : …`）。
5. **多言語はパスで分けない**。`/en/...` のような言語別ルートは作らない（現状はクライアント側 locale 切替）。
6. **ドキュメントを陳腐化させない**。機能・構成・コンポーネント・規約・デプロイ手順に影響する変更をしたら、対応する `docs/` ／ `README.md` ／ `SPEC.md` も**同じ変更で更新する**（下表「更新の目安」参照）。
7. **作業はコミットで区切ってから次に進む**。後から「やっぱり戻して」に即応できるよう、git をセーフポイントとして使う（下「開発の進め方」参照）。
8. **新ページは必ず SEO 3点セットを入れる**：`metadata`（title/description/openGraph）＋ `alternates: pageAlternates(path)`（canonical/hreflang）＋ JSON-LD（`webAppJsonLd`／記事は `articleJsonLd` ＋ `breadcrumbJsonLd`）。集客が目的なので必須。ヘルパーは `app/_lib/seo.ts`。
9. **プライバシー不変条件を壊さない**：バックエンド無し・ブラウザ内処理を維持し、**ユーザー入力をサーバー／外部に送信しない**。広告・解析以外の外部スクリプトや外部 API を安易に足さない（必要時は理由を明記して相談）。
10. **依存を安易に増やさない**：静的＆軽量が SEO・広告表示・表示速度に直結する。新しい npm 依存の追加は最小限にし、標準機能や既存ライブラリ（`marked`／`sonner` 等）で代替できないか先に検討する。

## 開発の進め方（コミット運用）

「後から特定の地点まで戻せる」状態を常に保つこと。**戻せる粒度＝コミットの粒度**。

- **新ツール／大きめの変更を始める前にブランチを切る**：`git switch -c tool/<名前>`。`main` を直接いじらない。
- **意味のある単位でコミットして区切る**。1コミットに複数の関心事を混ぜない（例：「盤面の骨組み」と「回転アニメ」は別コミット）。これにより「アニメだけ戻す」がピンポイントで可能になる。
- **新しいステップに進む前に、直前のステップをコミットする**。ユーザーが「さっきのは無し」と言ったら `git revert <コミット>`（履歴を残す安全な戻し方）で対応する。
- **`git reset --hard` や `git checkout -- <file>` など取り消しが効きにくい操作の前は、必ず `git log` で履歴を見せ、どこまで戻すかをユーザーに確認してから実行する**。勝手に巻き戻さない。
- コミット・ブランチ作成・プッシュは**ユーザーが指示したとき、または上記の区切りタイミングで**行う。`npm run build` が通ってからコミットするのを基本とする。

## やること / やらないこと

- ✅ **新ツール追加** は `docs/NEW_TOOL.md` のチェックリストに従う。記事と違い**自動生成されず**、`sitemap` / ホーム / `AppShell`（ナビ＋タイトルマップ）/ `RelatedTools` / `guide` / `TOOL_SHARE` / i18n辞書 への手動登録が必要（漏れるとSEO機会損失）。
- ✅ **記事追加** は `app/_lib/articles/<slug>.ts` を作成 → `app/_lib/articles.ts` に登録するだけ。本文は **Markdown**。ページ・一覧・サイトマップ・JSON-LD は自動生成。（`app/_lib/articles/README.md` と `docs/DEPLOY.md` 参照）
- ✅ **ツールページ** のヘッダー/フッターは共通の `ToolHeader` / `ToolFooter` を使う。
- ✅ **非公開モジュール** は `_` プレフィックスのフォルダ/ファイルに置く（例：`_components`, `_lib`, `_hooks`）。
- ✅ **SNSシェア文言** は `app/_lib/tools.ts` に集約。
- ❌ 描画用途（ルーレット盤面・あみだ線・OGP画像・アイコン生成）**以外**でハードコード色を増やさない。
- ❌ `node_modules` / `.next` を git に含めない（`.gitignore` 済み）。

## 参照ドキュメント & 保守ルール

コードを変更したら、**対応するドキュメントも同じ変更で更新する**こと（陳腐化させない）。

| ファイル | 内容 | この変更をしたら更新する |
|----------|------|--------------------------|
| `README.md` | 概要・起動/ビルド方法 | 技術スタック・概要・コマンドの変更 |
| `SPEC.md` | 全体仕様（機能・ページ・SEO・多言語） | 機能/ツール/ルート/SEO/多言語の変更 |
| `docs/ARCHITECTURE.md` | システム構成・ディレクトリ・描画モデル | ディレクトリ構成・描画方式・依存関係の変更 |
| `docs/COMPONENTS.md` | UIコンポーネント一覧 | 共通コンポーネントの追加/削除/役割変更 |
| `docs/STYLEGUIDE.md` | コーディング規約・デザイントークン | トークン・規約・命名・スタイルの変更 |
| `docs/DEPLOY.md` | デプロイ手順・記事追加手順 | デプロイ/環境変数/手順の変更 |
| `docs/NEW_TOOL.md` | 新ツール追加チェックリスト | 新ツール追加、登録箇所/手順の変更 |
| `docs/DEBUGGING.md` | デバッグ手順・修正の完了条件 | デバッグ手順/検証基準/再現方法の変更 |
| `docs/UPGRADING.md` | Next.js 等のメジャー/マイナー更新手順 | Next/React/Tailwind のバージョン更新・依存の大幅変更 |

> 補足：ドキュメントの文面修正だけのとき（コード無変更）は、この保守ルールの対象外（過剰更新を避ける）。
