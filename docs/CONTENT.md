# CONTENT.md — 記事制作・コンテンツ戦略ガイド

CastKit の記事（`app/_lib/articles/<slug>.ts`）を**何の目的で・どう書くか**の方針。
**追加の手順（ファイルの作り方・登録方法・Markdown記法）は `app/_lib/articles/README.md`** を参照。
このファイルは「中身（編集方針）」、README は「仕組み（手順）」と役割を分ける。

> AI（Claude 等）へ「記事を書いて」と頼むときは、まずこのファイルを参照させること。

---

## 0. 大原則：1記事 = 1検索意図

記事の単位は **ツールではなく「検索意図（キーワードのまとまり）」**。
「1ツール1記事」で止めず、**1つのツールに複数記事**をぶら下げてよい。
ただし条件は **「それぞれが別の検索意図を狙っていること」** ただ一つ。

- ❌ 悪い例（共食い＝カニバリゼーション）: 「お題ガチャの使い方」「お題ツールの使い方」「ランダムでお題を決める方法」
  → 検索する人が同じ。Google が評価を分散させ、**全部が中途半端に沈む**。
- ✅ 良い例（別意図）: 「雑談配信のネタ切れ解消（使い方）」「雑談のお題ネタ集（リスト）」「オンライン飲み会で盛り上がる質問（シーン特化）」
  → ターゲットが違うので共存し、すべて同じツールへ送客できる。

### 判定テスト（記事を分ける/まとめるの基準）

> **「この2記事を見つけるとき、人は“違う言葉”で検索するか？」**

- **違う言葉で検索する → 別記事にする（推奨）**
- **同じ言葉で検索する → 1記事にまとめる（分けると自滅）**

## 1. クラスター構造（ツール=ハブ、記事=枝）

各ツールページを**ハブ**、複数記事を**そこへ向かう枝**とする（ピラー＆クラスター型）。
記事は `toolHref` で対応ツールに紐づき、ツール⇄記事の相互リンクが自動生成される。
本文中でも関連ツール・関連記事へ内部リンクを張り、回遊と SEO 評価を高める。

記事タイプの目安：

| タイプ | 狙う層 | 検索ボリューム | 例 |
|--------|--------|----------------|-----|
| 使い方（navigational） | すでにツールを知っている人 | 小 | 「お題ガチャ 使い方」 |
| リスト/ネタ集（informational） | 悩み・ネタを探している人 | **大（集客の主力）** | 「雑談 お題 ネタ 一覧」 |
| シーン特化（informational） | 特定の場面で探す人 | 中 | 「オンライン飲み会 質問」 |

> 新サイトの集客は「リスト/ネタ集」「シーン特化」型を増やすことで伸びる。
> 「使い方」型だけだと、ツール名を既に知っている人にしか届かない。

## 2. 記事の標準構成

既存記事（`app/_lib/articles/*.ts`）のトーンに合わせる。本文は Markdown（`##`=h2 から。`#` は使わない）。

1. **導入（2〜4文）**：読者の悩みを言語化 → この記事で何が得られるか
2. **本文セクション（`##` 複数）**：検索意図に直接答える中身。リスト型なら具体例を惜しまず出す
3. **ツールへの送客セクション**：関連ツール（`toolHref`）の使いどころを `[テキスト](/topic)` 形式の内部リンクで案内
4. **関連機材（アフィリエイト）**：Amazon 直リンク `tag=castkit-22` を2〜3本。記事テーマに自然につながるものだけ（無理に押し込まない）
5. **まとめ**：要点 ＋ もう一度ツールへ誘導

### SEO フィールドの埋め方（`{ ja, en }` 両方必須）

- `title`：**検索キーワードを前方**に。例「雑談のお題・ネタ集｜…」
- `description`：100〜120字目安。キーワードを自然に含める
- `excerpt`：一覧カード用の短い紹介
- `slug`：英小文字＋ハイフン。ファイル名と一致
- `date`：公開日 `YYYY-MM-DD`（配列の並び順 = 一覧表示順。先頭が新着）

### 守るべき制約（CLAUDE.md 連動）

- **日英 両対応**（`{ ja, en }`）。英語は直訳でなく自然な表現に
- 描画用途以外で**色をハードコードしない**（記事本文は Markdown なので基本問題ない）
- **プライバシー不変条件**：本文や送客で外部にユーザー入力を送る導線を作らない
- アフィリエイトは Amazon 直契約 `tag=castkit-22`（もしも経由は現状未使用）

## 4. 記事を1本書くときの流れ（AI/人共通）

1. 対象ツールを決める → このファイル §3 で既存の検索意図を確認
2. **被らない検索意図を1つ**設定（§0 の判定テストを通す）
3. §2 の標準構成で日英の本文を書く（`_template.ts` をコピー）
4. `app/_lib/articles.ts` に登録（README 参照）
5. §3 のクラスターマップに行を追加
6. `npm run lint` & `npm run build` で検証（`/articles/<slug>` が生成されることを確認）
